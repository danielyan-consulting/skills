# 2025 CWE Top 25 Most Dangerous Software Weaknesses

Source: https://cwe.mitre.org/top25/archive/2025/2025_cwe_top25.html
Last updated: December 2025

Each entry below contains the CWE identifier, its official name, the MITRE-assigned
danger score, how many CVEs appear in CISA's Known Exploited Vulnerabilities (KEV)
catalogue, detection heuristics the agent should use when scanning source code, and
the recommended remediation approach.

---

## Rank 1 -- CWE-79: Improper Neutralisation of Input During Web Page Generation (Cross-site Scripting / XSS)
- **Score:** 60.38 | **KEV CVEs:** 7
- **Detection heuristics:**
  - User-controlled data rendered in HTML templates without escaping (e.g. `innerHTML`, `dangerouslySetInnerHTML`, `v-html`, `| safe`, `{% autoescape off %}`, `{!! !!}`, `<%= %>` without encoding).
  - String concatenation or interpolation that builds HTML from request parameters.
  - DOM manipulation with `document.write`, `element.outerHTML`, or jQuery `.html()` using untrusted values.
  - URL parameters reflected into `<script>` blocks, event-handler attributes, or `href="javascript:..."`.
  - Missing Content-Security-Policy headers in server configuration.
- **Remediation:**
  - Use context-aware output encoding (HTML-entity, JS, URL, CSS) appropriate to the insertion point.
  - Prefer frameworks that auto-escape by default (React JSX, Angular templates, Jinja2 autoescape).
  - Enforce a strict Content-Security-Policy that disallows inline scripts and `eval`.
  - Validate and sanitise all user input on the server side before storage or reflection.
  - Use allow-list validation for URLs and reject `javascript:` or `data:` schemes.

## Rank 2 -- CWE-89: Improper Neutralisation of Special Elements used in an SQL Command (SQL Injection)
- **Score:** 28.72 | **KEV CVEs:** 4
- **Detection heuristics:**
  - String concatenation or f-string / template-literal interpolation inside SQL query strings.
  - Use of `cursor.execute(f"SELECT ... {var}")` or equivalent without parameterised placeholders.
  - ORM `.raw()`, `.extra()`, `RawSQL`, or `Sequelize.literal()` with embedded user input.
  - Stored procedures built from concatenated strings.
  - Dynamic ORDER BY, table names, or column names derived from user input without allow-list validation.
- **Remediation:**
  - Use parameterised queries or prepared statements exclusively.
  - Use ORM query builders and avoid raw SQL wherever possible.
  - Apply allow-list validation for identifiers (table names, column names, sort directions) that cannot be parameterised.
  - Apply least-privilege database accounts -- the application should never connect as a superuser.
  - Deploy a Web Application Firewall (WAF) as a defence-in-depth layer.

## Rank 3 -- CWE-352: Cross-Site Request Forgery (CSRF)
- **Score:** 13.64 | **KEV CVEs:** 0
- **Detection heuristics:**
  - State-changing endpoints (POST, PUT, DELETE, PATCH) that lack CSRF token validation.
  - Forms without a CSRF hidden field or token header.
  - Framework CSRF middleware disabled or exempted (e.g. `@csrf_exempt`, `csrf: false`).
  - Cookie-based authentication without `SameSite` attribute.
  - API endpoints relying solely on cookies for authentication with no additional CSRF check.
- **Remediation:**
  - Enable and enforce framework-provided CSRF protection on all state-changing endpoints.
  - Use the Synchroniser Token Pattern or Double Submit Cookie pattern.
  - Set `SameSite=Strict` or `SameSite=Lax` on session cookies.
  - For APIs, prefer token-based authentication (Bearer tokens in headers) over cookie-based auth.
  - Require re-authentication or CAPTCHA for sensitive operations.

## Rank 4 -- CWE-862: Missing Authorisation
- **Score:** 13.28 | **KEV CVEs:** 0
- **Detection heuristics:**
  - Controller actions, API routes, or view functions that access or modify resources without checking the requesting user's permissions.
  - Endpoints that check authentication but not authorisation (i.e. any logged-in user can perform any action).
  - Missing role checks or permission guards on admin, management, or privileged routes.
  - Direct object references where the code fetches a record by ID from user input without verifying ownership.
  - Absence of authorisation middleware or decorators on route definitions.
- **Remediation:**
  - Implement and enforce authorisation checks on every endpoint, not just authentication.
  - Use a centralised authorisation framework (RBAC, ABAC, or policy engine such as OPA / Casbin / CASL).
  - Default-deny: require explicit permission grants rather than implicit access.
  - Write integration tests that verify unauthorised users receive 403 responses.
  - Audit all routes and handlers for missing authorisation decorators or middleware.

## Rank 5 -- CWE-787: Out-of-bounds Write
- **Score:** 12.68 | **KEV CVEs:** 12
- **Detection heuristics:**
  - Array or buffer writes where the index or length is derived from user input without bounds checking.
  - Use of unsafe C/C++ functions: `strcpy`, `strcat`, `sprintf`, `gets`, `memcpy` without length validation.
  - Manual memory management with `malloc`/`calloc` followed by writes that may exceed the allocated size.
  - Off-by-one errors in loop bounds that write one element past the end of an array.
  - Pointer arithmetic without corresponding bounds verification.
- **Remediation:**
  - Use bounds-checked alternatives: `strncpy`, `strncat`, `snprintf`, or safer wrappers.
  - Prefer memory-safe languages (Rust, Go, Java, C#) for new development.
  - Enable compiler protections: stack canaries, ASLR, and `-D_FORTIFY_SOURCE=2`.
  - Use static analysis tools (e.g. Coverity, CodeQL) and address sanitisers during testing.
  - Validate all buffer sizes and indices against allocation sizes before writing.

## Rank 6 -- CWE-22: Improper Limitation of a Pathname to a Restricted Directory (Path Traversal)
- **Score:** 8.99 | **KEV CVEs:** 10
- **Detection heuristics:**
  - User-supplied file names or paths passed directly to file-system operations (`open`, `readFile`, `send_file`, `os.path.join` with untrusted segments).
  - Failure to canonicalise paths before checking they fall within an allowed directory.
  - Patterns like `../` or `..\\` not stripped or rejected from user input used in file paths.
  - Archive extraction (zip, tar) without validating member paths (Zip Slip).
  - Serving static files where the request URL is mapped directly to the file system.
- **Remediation:**
  - Canonicalise (resolve) the full path and verify it starts with the intended base directory.
  - Use allow-list validation for permitted file names or extensions.
  - Reject any input containing `..`, null bytes, or path separator characters.
  - Use chroot, containers, or sandboxed file-system access to limit reachable paths.
  - When extracting archives, validate every member path before extraction.

## Rank 7 -- CWE-416: Use After Free
- **Score:** 8.47 | **KEV CVEs:** 14
- **Detection heuristics:**
  - Pointers or references used after the memory they point to has been freed or deallocated.
  - Double-free patterns where `free()` or `delete` is called more than once on the same pointer.
  - Returning pointers to stack-allocated or freed heap memory from functions.
  - Event callbacks or closures that capture references to objects whose lifetime has ended.
  - Shared ownership without reference counting or smart pointers in C/C++.
- **Remediation:**
  - Use smart pointers (`std::unique_ptr`, `std::shared_ptr`) instead of raw pointers in C++.
  - Set pointers to `NULL`/`nullptr` immediately after freeing.
  - Prefer memory-safe languages for new development.
  - Use AddressSanitizer (ASan) during testing to detect use-after-free at runtime.
  - Apply static analysis tools that track object lifetimes.

## Rank 8 -- CWE-125: Out-of-bounds Read
- **Score:** 7.88 | **KEV CVEs:** 3
- **Detection heuristics:**
  - Array or buffer reads where the index is derived from user input without bounds checking.
  - Loop conditions that may read beyond the allocated buffer.
  - String functions operating on buffers without guaranteed null termination.
  - Parsing binary formats where field lengths are taken from the data without validation.
  - Pointer arithmetic used to compute read offsets without verifying the result stays within bounds.
- **Remediation:**
  - Always validate indices and lengths against buffer sizes before reading.
  - Use bounds-checked container types (e.g. `std::vector::at()` in C++, slices in Go/Rust).
  - Enable compiler and runtime protections (ASan, bounds-checking modes).
  - Prefer memory-safe languages for parsing untrusted binary data.
  - Fuzz-test all parsers that consume external data.

## Rank 9 -- CWE-78: Improper Neutralisation of Special Elements used in an OS Command (OS Command Injection)
- **Score:** 7.85 | **KEV CVEs:** 20
- **Detection heuristics:**
  - Use of `os.system()`, `subprocess.Popen(shell=True)`, `exec()`, `Runtime.exec()`, backtick operators, or `child_process.exec()` with user-supplied data.
  - String concatenation or interpolation that builds shell command strings from user input.
  - Piping user input into commands without sanitisation.
  - CGI scripts or server handlers that invoke system commands with request parameters.
  - Use of `eval()` or similar constructs to execute system commands dynamically.
- **Remediation:**
  - Avoid invoking OS commands from application code wherever possible; use libraries or APIs instead.
  - When OS commands are unavoidable, use parameterised APIs (e.g. `subprocess.run([...], shell=False)` with argument lists).
  - Apply strict allow-list validation on any user input that must form part of a command.
  - Run the application with least-privilege OS accounts.
  - Use sandboxing (seccomp, AppArmor, SELinux) to restrict callable executables.

## Rank 10 -- CWE-94: Improper Control of Generation of Code (Code Injection)
- **Score:** 7.57 | **KEV CVEs:** 7
- **Detection heuristics:**
  - Use of `eval()`, `exec()`, `Function()`, `setTimeout(string)`, `setInterval(string)` with user-controlled input.
  - Template engines configured to allow arbitrary code execution (e.g. SSTI in Jinja2, Mako, Freemarker, Pug).
  - Dynamic imports, reflection, or class loading from user-supplied strings.
  - Server-side rendering that compiles user input as code.
  - Build or deployment scripts that interpolate environment variables or user input into executable code.
- **Remediation:**
  - Never pass user input to code evaluation functions.
  - Use sandboxed template engines with restricted execution contexts.
  - Prefer data-driven logic over dynamically generated code.
  - If dynamic code is absolutely required, run it in a sandboxed environment (e.g. V8 isolate, WebAssembly sandbox).
  - Apply input validation and reject any input containing code-like syntax.

## Rank 11 -- CWE-120: Buffer Copy without Checking Size of Input (Classic Buffer Overflow)
- **Score:** 6.96 | **KEV CVEs:** 0
- **Detection heuristics:**
  - Use of `strcpy`, `strcat`, `sprintf`, `gets`, `scanf("%s", ...)` in C/C++.
  - Copying user-controlled data into fixed-size buffers without length checks.
  - Missing size parameter in memory-copy operations.
  - Functions that accept a buffer pointer but no corresponding size argument.
  - Wrappers around unsafe copy functions that do not add bounds validation.
- **Remediation:**
  - Replace all unsafe copy functions with bounded variants (`strncpy`, `strlcpy`, `snprintf`, `fgets`).
  - Use `std::string`, `std::vector`, or equivalent managed types in C++.
  - Enable `-fstack-protector-all`, ASLR, and DEP/NX at compile and link time.
  - Run static analysis and fuzz testing on all C/C++ code.
  - Migrate critical components to memory-safe languages where feasible.

## Rank 12 -- CWE-434: Unrestricted Upload of File with Dangerous Type
- **Score:** 6.87 | **KEV CVEs:** 4
- **Detection heuristics:**
  - File upload endpoints that accept files without validating content type, extension, or magic bytes.
  - Uploaded files stored within the web root where they can be directly accessed and executed.
  - File type validation based solely on the `Content-Type` header or file extension (both client-controlled).
  - Absence of server-side content inspection (magic byte / MIME sniffing).
  - Missing restrictions on uploadable file size.
- **Remediation:**
  - Validate file type on the server using magic bytes, not just extensions or MIME headers.
  - Store uploaded files outside the web root or in a dedicated object store (S3, GCS).
  - Rename uploaded files with random, non-guessable names and strip original extensions.
  - Serve user-uploaded content from a separate domain with restrictive Content-Type headers.
  - Enforce file size limits and scan uploads for malware.

## Rank 13 -- CWE-476: NULL Pointer Dereference
- **Score:** 6.41 | **KEV CVEs:** 0
- **Detection heuristics:**
  - Dereferencing pointers or object references without prior null checks.
  - Functions that may return `NULL`/`null`/`None`/`nil` whose return value is used without verification.
  - Error-handling paths that neglect to check for null before using a result.
  - Optional values unwrapped without guards (e.g. forced unwrap `!` in Swift, `.unwrap()` in Rust on a `None`).
  - Chained method calls on potentially null objects without null-safe operators.
- **Remediation:**
  - Check for null before every dereference of values that may be null.
  - Use option/maybe types and pattern matching instead of nullable pointers.
  - Enable compiler warnings for potential null dereferences (`-Wnull-dereference`).
  - Use static analysis to detect unchecked null paths.
  - Adopt null-safety features in languages that support them (Kotlin, Dart, C# nullable reference types).

## Rank 14 -- CWE-121: Stack-based Buffer Overflow
- **Score:** 5.75 | **KEV CVEs:** 4
- **Detection heuristics:**
  - Large or variable-length arrays allocated on the stack (e.g. `char buf[MAX_INPUT]` where MAX_INPUT is large or user-influenced).
  - `alloca()` called with user-controlled sizes.
  - Recursive functions with deep or unbounded recursion depth.
  - Local buffers receiving data from network or file input without size checks.
  - Functions with variable-length argument lists processed into stack buffers.
- **Remediation:**
  - Allocate large or variable-size buffers on the heap with explicit size management.
  - Use stack canaries and compiler protections (`-fstack-protector-strong`).
  - Limit recursion depth and prefer iterative algorithms where possible.
  - Validate all input lengths before copying into stack buffers.
  - Use AddressSanitizer and StackSanitizer during development and CI.

## Rank 15 -- CWE-502: Deserialisation of Untrusted Data
- **Score:** 5.23 | **KEV CVEs:** 11
- **Detection heuristics:**
  - Use of `pickle.loads()`, `yaml.load()` (without `SafeLoader`), `Marshal.load`, Java `ObjectInputStream`, PHP `unserialize()`, .NET `BinaryFormatter` on untrusted data.
  - JSON deserialisation with polymorphic type handling enabled (e.g. Jackson `enableDefaultTyping`, `JsonTypeInfo`).
  - Message queues or RPC endpoints that deserialise objects from untrusted sources.
  - Cookie values or hidden form fields containing serialised objects.
  - Custom binary protocols that reconstruct objects from wire data without validation.
- **Remediation:**
  - Never deserialise data from untrusted sources using native serialisation formats.
  - Use safe data-exchange formats (JSON, Protocol Buffers, MessagePack) with strict schemas.
  - If native deserialisation is required, use type allow-lists and disable polymorphic type resolution.
  - Implement integrity checks (HMAC, digital signatures) on serialised data.
  - Monitor and alert on deserialisation errors as potential attack indicators.

## Rank 16 -- CWE-122: Heap-based Buffer Overflow
- **Score:** 5.21 | **KEV CVEs:** 6
- **Detection heuristics:**
  - `malloc`/`calloc`/`realloc` followed by writes that exceed the allocated size.
  - Heap-allocated buffers resized based on user input without validation.
  - Off-by-one errors in heap buffer calculations.
  - Copying data into heap buffers using unsafe functions (`memcpy`, `strcpy`) with incorrect size.
  - Custom memory allocators without overflow checks on size arithmetic.
- **Remediation:**
  - Validate all sizes before allocation and all lengths before writes.
  - Use safe allocation wrappers that check for integer overflow in size calculations.
  - Prefer `std::vector`, `std::string`, or equivalent managed heap types.
  - Enable HeapSanitizer and fuzz testing in CI pipelines.
  - Migrate to memory-safe languages for new components.

## Rank 17 -- CWE-863: Incorrect Authorisation
- **Score:** 4.14 | **KEV CVEs:** 4
- **Detection heuristics:**
  - Authorisation logic that uses client-supplied roles, permissions, or user IDs without server-side verification.
  - Flawed conditional checks (e.g. `if (role != "admin")` instead of `if (role == "admin")`).
  - Authorisation decisions based on obscurity (hidden URLs, undocumented API endpoints).
  - Middleware that checks permissions but fails open on error.
  - Inconsistent authorisation enforcement across different code paths to the same resource.
- **Remediation:**
  - Centralise authorisation logic and apply it uniformly at the middleware/framework level.
  - Use a policy engine with well-tested, declarative rules.
  - Default-deny: deny access unless explicitly granted.
  - Write comprehensive authorisation unit and integration tests, including negative test cases.
  - Log and monitor authorisation failures for anomaly detection.

## Rank 18 -- CWE-20: Improper Input Validation
- **Score:** 4.09 | **KEV CVEs:** 2
- **Detection heuristics:**
  - User input consumed without any validation, sanitisation, or type checking.
  - Numeric inputs not checked for range, sign, or overflow.
  - String inputs not checked for length, format, or character set.
  - Input used to make trust decisions (e.g. IP-address allow-lists relying on `X-Forwarded-For`).
  - Catch-all or permissive input acceptance patterns (e.g. regex `.*`).
- **Remediation:**
  - Validate all inputs against strict schemas: type, length, range, format, and character set.
  - Use allow-list validation where the set of valid inputs is known.
  - Validate on the server side regardless of client-side validation.
  - Use strongly-typed APIs and data-transfer objects to enforce constraints.
  - Reject and log invalid input rather than attempting to sanitise it into a valid form.

## Rank 19 -- CWE-284: Improper Access Control
- **Score:** 4.07 | **KEV CVEs:** 1
- **Detection heuristics:**
  - Resources (files, database records, API endpoints) accessible without any access-control mechanism.
  - Access-control checks present in the UI but absent from the back-end.
  - Horizontal privilege escalation: user A can access user B's resources by modifying an identifier.
  - Vertical privilege escalation: a regular user can access administrative functions.
  - Default or overly permissive permissions on files, directories, or cloud resources.
- **Remediation:**
  - Apply principle of least privilege at every layer (application, database, OS, cloud).
  - Enforce access control on the server side for every request.
  - Use role-based or attribute-based access control with a centralised policy.
  - Regularly audit permissions and access patterns.
  - Test for both horizontal and vertical privilege escalation in security reviews.

## Rank 20 -- CWE-200: Exposure of Sensitive Information to an Unauthorised Actor
- **Score:** 4.01 | **KEV CVEs:** 1
- **Detection heuristics:**
  - Error messages, stack traces, or debug output returned to the client in production.
  - Sensitive data (passwords, API keys, tokens, PII) logged at INFO or DEBUG level.
  - API responses that include more fields than the client requires (over-fetching).
  - Comments in source code or HTML that reveal internal architecture, credentials, or algorithms.
  - Sensitive data transmitted without encryption or stored in plaintext.
- **Remediation:**
  - Return generic error messages to clients; log detailed errors server-side only.
  - Audit all API responses and strip unnecessary fields; use allow-list serialisation.
  - Remove debug code, comments containing sensitive information, and verbose logging from production.
  - Encrypt sensitive data in transit (TLS) and at rest.
  - Use secrets management tools; never commit credentials to source control.

## Rank 21 -- CWE-306: Missing Authentication for Critical Function
- **Score:** 3.47 | **KEV CVEs:** 11
- **Detection heuristics:**
  - Administrative, management, or sensitive endpoints that do not require any authentication.
  - API routes that perform privileged operations but have no auth middleware.
  - Internal microservice endpoints exposed to the network without authentication.
  - Configuration or health-check endpoints that reveal sensitive data without auth.
  - Default installations with open admin panels or management consoles.
- **Remediation:**
  - Require authentication on every endpoint that accesses or modifies sensitive resources.
  - Apply authentication at the gateway or load-balancer level as a defence-in-depth measure.
  - For internal services, use mutual TLS (mTLS) or service mesh authentication.
  - Audit all routes and verify each has explicit authentication requirements.
  - Disable or secure default admin panels and management interfaces.

## Rank 22 -- CWE-918: Server-Side Request Forgery (SSRF)
- **Score:** 3.36 | **KEV CVEs:** 0
- **Detection heuristics:**
  - Server-side code that fetches URLs or resources specified by user input (e.g. `requests.get(user_url)`, `fetch(userUrl)`, `curl_exec` with user-supplied URL).
  - Webhook, callback, or integration endpoints that accept arbitrary URLs.
  - Image/PDF renderers or link-preview generators that fetch user-supplied URLs.
  - URL parameters used to specify redirect targets, proxy destinations, or API endpoints.
  - Cloud metadata endpoints (169.254.169.254) not blocked in outbound request validation.
- **Remediation:**
  - Validate and restrict destination URLs against a strict allow-list of permitted hosts and schemes.
  - Block access to internal/private IP ranges (10.x, 172.16-31.x, 192.168.x, 169.254.x, localhost).
  - Use a dedicated egress proxy that enforces URL allow-lists.
  - Disable unnecessary URL schemes (file://, gopher://, dict://).
  - Run server-side HTTP clients with network policies that prevent access to internal infrastructure.

## Rank 23 -- CWE-77: Improper Neutralisation of Special Elements used in a Command (Command Injection)
- **Score:** 3.15 | **KEV CVEs:** 2
- **Detection heuristics:**
  - User input embedded into command strings for databases, LDAP, XML processors, or other interpreters.
  - Template strings, format strings, or concatenation used to build commands for any interpreter.
  - Failure to escape or parameterise special characters meaningful to the target interpreter.
  - Dynamic construction of LDAP queries, XPath expressions, or NoSQL queries from user input.
  - Use of `eval`-like constructs in shell, Perl, Ruby, or other scripting languages.
- **Remediation:**
  - Use parameterised or structured APIs that separate code from data for every interpreter.
  - Apply context-specific escaping for the target command language.
  - Use allow-list validation on any user input that forms part of a command.
  - Avoid invoking external interpreters; prefer native-language libraries.
  - Apply least-privilege principles to interpreter accounts and permissions.

## Rank 24 -- CWE-639: Authorisation Bypass Through User-Controlled Key
- **Score:** 2.62 | **KEV CVEs:** 0
- **Detection heuristics:**
  - Database lookups using record IDs supplied directly by the user without ownership verification (Insecure Direct Object References / IDOR).
  - APIs that accept a `user_id`, `account_id`, or similar parameter and return data without checking that the authenticated user owns that record.
  - Sequential or predictable identifiers (auto-increment IDs) used as access keys.
  - Endpoints that allow users to modify resources by specifying the resource ID without verifying the caller's relationship to that resource.
  - Multi-tenant applications where tenant isolation relies solely on client-supplied tenant IDs.
- **Remediation:**
  - Always verify server-side that the authenticated user is authorised to access the requested resource.
  - Use indirect references: map opaque tokens to internal IDs server-side.
  - Replace sequential IDs with UUIDs or other non-guessable identifiers.
  - Implement and test ownership checks for every data-access operation.
  - Apply tenant-scoping at the data-access layer so queries automatically filter by authenticated tenant.

## Rank 25 -- CWE-770: Allocation of Resources Without Limits or Throttling
- **Score:** 2.54 | **KEV CVEs:** 0
- **Detection heuristics:**
  - Unbounded loops, recursion, or data structures whose size is controlled by user input.
  - Missing rate limiting on API endpoints, login attempts, or resource-intensive operations.
  - File uploads, batch processing, or query endpoints with no size or count limits.
  - Thread or process creation driven by user requests without pooling or caps.
  - Memory allocations sized by user input without upper-bound checks.
- **Remediation:**
  - Enforce rate limiting and request throttling on all public-facing endpoints.
  - Set explicit upper bounds on all resource allocations (memory, threads, connections, file size, query results).
  - Use resource quotas, cgroups, or cloud resource policies to cap consumption.
  - Implement circuit breakers and backpressure mechanisms.
  - Monitor resource usage and alert on anomalous consumption patterns.
